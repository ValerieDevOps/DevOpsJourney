Logs are time-stamped records of events generated by operating systems, services, applications, users, and security tools. 

Type of logs
System logs record operating system and service events. 
Application logs record application behaviour. 
Security logs record authentication events and suspicious activity.

Log levels are categories / labels that help you determine the severity/importance of a message


Log Levels
DEBUG – detailed troubleshooting information
INFO – normal operational events
WARNING – unusual conditions
ERROR – failures requiring attention
CRITICAL – severe system failures

SSH means Secure Shell and it is a way to connect or control another computer or server over a network

Start SSH Service
sudo systemctl start ssh

Check Status of Service
sudo systemctl status ssh

Read, Display only SSH log and follow it in real time
sudo journalctl -u ssh -f

Server machine (the system you want to access)
This is where you start SSH service
This is your “target system”


Client machine (the system you connect FROM)
This is where you open another terminal
This is where you run the SSH command

LOG IN TO MY SYSTEM FROM ANOTHER SYSTEM (SERVER ACCESS)

1. Start SSH service (ON MY SYSTEM / SERVER)
   sudo systemctl start ssh

2. Check status of SSH service
   sudo systemctl status ssh

3. From another system (CLIENT MACHINE), open terminal and connect:
   ssh my_username@server_ip

   Example:
   ssh valerie@192.168.1.10


 LOGIN FROM MY SYSTEM TO ANOTHER SYSTEM (CONNECT TO REMOTE SERVER)

1. Start SSH service (ON THE TARGET SERVER SYSTEM)
   sudo systemctl start ssh

2. Check status of SSH service
   sudo systemctl status ssh

3. From my system (CLIENT MACHINE), open terminal and connect:
   ssh client_username@client_server_ip_address

   Example:
   ssh admin@10.0.0.5


START SSH SERVICE ON MY SYSTEM + TEST LOCAL LOGIN

1. Start SSH service
   sudo systemctl start ssh

2. Check status of SSH service
   sudo systemctl status ssh

3. Open another terminal and test local SSH:
   ssh localhost

   OR
   ssh my_username@localhost

IMPORTANT RULES

# - "server_username" = username on the SERVER (not client)
# - SSH service runs ONLY on the server
# - Client

Failed login attempts is an attack that can be investigated using SSH service.

Simulating a failed in attempt
Start SSH Service
Check the status of the service
3. Open another terminal and try to login
 
ssh fakeuser@localhost

4 Enter password (a wrong password) continuously

Investigate a failed login Attempt

First Way

Set up the command to display your log

sudo journalctl -u ssh

2. Filter for "Failed Password" from your log in your authentication files
sudo grep "Failed password" /var/log/auth.log

Second Way
Combine both the command to display and filter logs from your ssh service not authentication logs.

ssh journalctl -u ssh | grep "Failed Password"

NB: Authentication Logs  shows log entries across all services not just a service

Simple Solution:
 1. Restart the service 
sudo systemctl restart ssh 

2. Check the status of SSH service
sudo systemctl status ssh

3. Open another terminal and  re-login after confirming th right password to the system user account you are trying to access.

4. Verify journald is running and retaining records.

Logging must remain enabled. A server without logs is nearly impossible to investigate.

type; sudo systemctl status systemd-journald

5. Check the disk usage of journald

sudo journalctl --disk-usage
